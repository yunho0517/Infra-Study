[ Ping이 동작하는 과정 ]
ping 전에 ARP가 먼저 발생한다.
PC1은 192.168.1.10이라는 IP를 알고 있지만, MAC 주소는 모르고 있다.
그래서 ARP Request를 먼저 보내어 MAC 주소를 확인한다.
그 이후에 Ping이 정상적으로 진행이 된다.

1. ARP로 MAC 주소를 조회한다.
2. ICMP Echo Request를 전송한다
3. 상대방이 ICMP Echo Request를 반환한다.
4. Ping을 진행한다.

[ VLAN의 존재 이유! ]
*  먼저 개발팀, 인사팀, 회계팀 이렇게 3개의 팀이 있다고 가정할 때 방금처럼 하나의 스위치에
   모두 연결하면 전부 통신이 가능하기는 하다.
   하지만, 개발팀이 회계팀 또는 인사팀에,  인사팀이 개발팀 또는 회계팀에 접근이 가능하기 때문에
   보안상 좋지 않은 방식이다.

   그래서 등장한 것이 VLAN (Virtual LAN) 이다. 즉, 가상 네트워크다.
   예를 들어, 스위치는 하나지만, 논리적으로 VLAN 10 = 개발 / VLAN 20 = 인사 이렇게 나누는 것
   이렇게 나누면, 같은 스위치에 꽂혀 있어도 서로 다른 네트워크처럼 행동하게 된다.




[ 회사 네트워크 구축해보기 ]
1. 스위치 -> CLI창 -> enable 입력하여 switch# 상태로 만들기

2. 설정 모드로 가기 위해 configure terminal 입력 (conf t : 축약버전으로도 가능)
   ㄴ> "이제부터 스위치 설정을 변경하겠다.

3. 프롬프트가 swtich(config)#으로 변한것을 확인할 수 있음.

4. vlan 10 : VLAN 10 생성 (번호가 10번인 VLAN을 생성)

5. switch(config-vlan)# 으로 프롬프트가 변경됨.

6. name DEV : VLAN 10 이름을 DEV로 지정. (확인하기 쉽도록 붙이는 것)

7. exit : 빠져나오기

8. 같은 방식으로 VLAN 20 생성 (인사팀)

9. exit로 프롬프트가 switch# 으로 바뀐 것을 확인

10. show vlan brief : VLAN 들을 보여준다. (테이블 형식으로)

11. conf t 입력 -> interface range fa0/1 - 2 : Fa0/1 과 Fa0/2 두 포트를 동시에 선택한 것

12. switchport mode access : 일반 PC 연결용 포트

13. switchport access vlan 10 : 이 포트들은 VLAN10 소속으로 넣겠다.

14. exit -> interface range fa0/3 - 4 

15. switchport mode access -> switchport access vlan 20

16. exit -> exit -> show vlan brief 했을 때 vlan 별로 포트가 지정되었음을 확인할 수 있다.

17. PC0 에서 ping 192.168.1.20을 하게 되면 정상적으로 연결되는 것을 확인할 수 있는데
    이것은 PC0과 PC1이 같은 VLAN 10 포트를 사용하고 있기 때문이다
    즉, 같은 네트워크를 사용하고 있다고 지정했기 때문에 가능하다.
    하지만 PC3나 PC4로는 실패한다. 이유는 두 PC는 VLAN 20 포트를 사용하고 있기 때문이다.
    즉, 같은 스위치지만, VLAN으로 인해 다른 네트워크처럼 분리되었기 때문에 발생한다.

    이는 곧, IP가 같다고 무조건 통신이 된다는 것은 아니라는 것이다.
    VLAN이 다르면 IP가 같아도 통신이 되지 않는다.




[ Inter-VLAN Routing ]
* 보통 회사에서 네트워크를 구현할때는 각 팀마다 이렇게 네트워크를 구분해서 하지만,
  가끔 개발부에서 영업부로의 통신이 필요하거나 그 반대의 상황이 생길때가 있다.
  이 경우 VLAN 포트 즉, 네트워크의 영역이 서로 달라 통신이 불가한데, 이것을 해결하는 것이
  라우터이다.
  라우터를 스위치에 연결하여 서로 다른 네트워크끼리도 통신이 가능하게 할 수 있다.

1. Router 2911을 스위치 위에 둔다.

2. Straight 선을 스위치에 연결한다 (Fa 0/24)

3. 초록불 될 때까지 대기.

*** 왜 포트 하나만 연결하는가?
	ㄴ> VLAN이 2개기에 포트를 2개 연결해야 한다고 생각하지만, 실무에서는 Trunk를 사용한다
    	      즉, 한 가닥으로 여러 VLAN을 전달할 수 있기 때문에 굳이 많이 연결하지 않는다.

4. 스위치 CLI 진입 -> enable 입력

5. conf t -> interface fa0/24 입력 

6. switchport mode trunk : 여러 VLAN을 동시에 통과시켜라

7. exit 이후 show interface trunk --> Fa0/24가 보여야함

8. 라우터 CLI 진입 -> enable 입력

9. conf t -> interface g0/0 -> no shutdown : 라우터는 기본적으로 꺼져있기 때문에 켜야함.

10. exit 이후 interface g0/0.10 : VLAN10 담당으로 지정 (VLAN10용 인터페이스)

11. encapsulation dot1Q 10 : VLAN10 태그 처리 

12. ip address 192.168.10.1 255.255.255.0 입력 

13. 같은 방식으로 VLAN20용 인터페이스를 만들기 

14. 하지만 아직 서로 다른 네트워크 간의 통신은 불가함. 이를 해결하기 위해 Gateway가 있어야 함
     예를 들어 192.168.10.10 ---> 192.168.20.10 으로 보내려 하지만 같은 네트워크가 아니기 때문에
     이를 보내지 못함. 그렇다면 이것을 같은 네트워크가 아니라고 생각하고 라우터에게로 보내게 됨
     그리고 그 라우터 주소가 192.168.10.1 이 되는 것임.

     즉, PC0 -> switch - Router -> swtich -> PC2 로 흐름이 이어짐.
    
     따라서 두 과정은 라우터가 각각 VLAN10과 VLAN20의 동일 네트워크를 가지고 있어야
     중간에서 전달해 줄 수 있기 때문에 인터페이스를 만든 것임.


15. 각 PC의 IP를 192.168.10.10, 20  /  192.168.20.10, 20 으로 바꾼뒤
     게이트웨이를 VLAN10 그룹은 192.168.10.1 로 입력하고
     게이트웨이를 VLAN20 그룹은 192.168.20.1 로 입력하기

16. 이후 PC0에서 PC2로의 ping 테스트를 하면 성공하는 것을 볼 수 있음.

17. 이때 처음 핑을 보내는 PC (다른 네트워크)는 첫시도는 실패, 나머지 3번은 성공하게 되는데
     그 이유는 ARP 때문이다.
     PC가 다른 네트워크에게 전달하기 위해 게이트웨이로 패킷을 보내게 되는데 문제는
     IP 주소는 알고 있지만, 게이트웨이의 MAC 주소를 모르고 있다.
     따라서 먼저 ARP 요청이 발생하고, 그 과정에서 첫번째 시도가 실패 발생 가능성이 생긴다.



< 오늘 중요 내용 >
     * VLAN10과 VLAN20은 왜 통신이 안되나요?
	ㄴ> VLAN은 별도의 브로드캐스트 도메인이기 때문이다.

     * 그렇다면 어떻게 통신시킬 수 있나요?
	ㄴ> Router 또는 L3 Switch(라우터 스위치)를 이용한 Inter-VLAN Routing이 필요하다.






[ 서브넷팅 ]
192.168.10.10 주소와
255.255.255.0 을 주로 위의 실습에서 썼는데 이를 표현하면

이 설정은 192.168.10.0/24 와 같은 의미이다.
그리고 IP는 총 32비트로 구성되어 있는데 그중에서
앞 24비트 = 네트워크
뒤 8비트 = 호스트 
이런 의미들을 가지고 있다.

따라서 192.168.10 까지는 네트워크이고 그 뒤에 10, 20, 30, 40 부분은 컴퓨터 번호라고 본다.

그렇다면 왜 서브넷팅을 할까?
	ㄴ> 회사에 PC가 10대 밖에 없는데, 매번 255.255.255.0을 사용하면
	     가능한 주소 수는 254개 (256개 중 1개는 브로드캐스트, 1개는 네트워크 주소)
	     즉, 주소의 낭비가 엄청나게 된다.
	     그렇기 때문에 부서 수에 맞게 네트워크의 크기를 줄인다.


1. 이전 실습에서는 192.168.10.0/24 를 사용하였다면 이번에는 그것보다 주소의 개수를 줄인
   192.168.10.0/28로 사용할 것이다.
   (2^32-28 = 2^4 = 16)
   즉, 이는 28비트를 네트워크로 사용하고 남은 4비트를 호스트로 사용하는 것이다.
   따라서 16개의 주소를 사용할 수 있는데, 이때 2개는 사용하지 못한다.
   가장 첫 주소인 192.168.10.0 은 네트워크 주소이고
   가장 마지막 주소인 192.168.10.15 는 브로드캐스트 주소이기 때문에 실제 가능한 것은
   이 두개를 뺀 14개의 주소이다.

2. PC0과 PC1의 주소를 192.168.10.10, 11 / 255.255.255.240으로 변경한다
   이때 240인 이유는 /28로 변경할 것이기 때문에 마지막 옥텟이 11110000으로 되게 된다.
   그리고 이것이 240이 되기 때문에 240으로 변경한 것이다.

3. 라우터의 VLAN10 인터페이스도 기존에서 255.255.255.240으로 변경시킨다.
   이때 이미 IP가 있으면 
   ' no ip address '라고 먼저 입력 후 해야한다.

4. conf t -> interface g0/0.10 -> no ip address -> ip address 192.168.10.1 255.255.255.240

5. 이때 PC1의 IP주소를 192.168.10.20으로 바꾸게 되는 경우 ping 명령어가 실패하게 되는데
   이유는 우리가 아까 /28로 바꿨다. (CIDR 표기)
   즉, 주소의 범위가 192.168.10.1 ~ 192.168.10.14 이기 때문에 해당 범위에 들어가지 않게 된다.
   이로 인해 통신 문제가 발생하게 된다.
   허나 패킷 트레이서가 라우터와 연결이 되어 있기 때문에 현재 상태에서는 통신 문제는 없다.





[ 패킷 트레이서의 Simulation Mode ]
1. 우측 하단에 Simulation 버튼 클릭

2. Reset simulation 을 눌러 기존 패킷 제거

3. 필터 설정 (Edit Filters) -> 많은 프로토콜 중에 ICMP 와 ARP만 활성화

4. PC0 -> ping 192.168.20.10 

5. Event List에 생긴것을 확인하고 이제 한 단계씩 과정을 추적해볼 것이다.

6. Capture/Forward (앞으로 재생) 클릭하여 과정을 본다

아마 첫 과정은 ARP Request 일 가능성이 높다 (처음 연결하는 경우)
왜냐하면 PC0은 게이트웨이의 MAC주소를 모르기 때문에 물어봐야 한다.

7. 화면에 보이는 이미지를 클릭해보면 Layer 1 ~ 7까지 순서대로 정렬된 것을 볼 수 있는데
   ARP는 보통 Layer 2 중심이다.
   이 부분이 MAC 주소를 찾는 프로토콜이기 때문이다.

8. 스위치는 ARP Request를 받을때 Flood 한다.
    즉, 모든 포트로 뿌린다.
   그 이유는 아직 어디가 목적지 MAC인지 모르기 때문이다.

9. 이러한 과정을 통해서 모든 프로토콜... 즉, 데이터 패킷이 통신할 때에는
    PC0 -> Switch -> Router -> Switch -> 목적지 -> Switch -> Router -> Switch -> PC0 으로 된다



<중요 내용>
1. Ping은 어떤 프로토콜을 사용하나요?
	ㄴ> ICMP를 사용한다

2. Ping이 성공하면 TCP도 성공한 건가요?
	ㄴ> ICMP만 성공한 것이다. TCP 서비스는 별도로 확인해야 한다.

3. 라우터를 통해서 다른 네트워크로 통신을 한다면, 해당 목적지의 MAC주소는 몰라도 된다.
   단지 우리가 알아야 할 것은 바로 다음 목적지의 MAC주소를 알아야 하는 것이다.
   즉, 게이트웨이의 MAC주소만 알면 된다. 
 
4. MAC주소는 라우터를 지날때마다 바뀐다.

5. 즉, PC가 다른 네트워크의 서버에 접속할 때 ARP는 게이트웨이의 MAC주소를 알아내기 위해
   ARP를 수행한다.

6. 라우터를 통과할때 무엇이 바뀌고 무엇이 유지됩니까?
	ㄴ> IP 주소는 유지되고, Layer2의 MAC 주소는 홉(Hop)마다 새로 변경된다.
 	     라우터는 목적지 IP를 기반으로 경로를 결정한다.
